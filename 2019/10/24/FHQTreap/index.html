为什么一定要转呢？（FHQ的拷问）

原理

和 Treap 一样，FHQTreap 维护两个值 key 和 val，key 是每个点的随机权值，满足堆性质，val 是每个节点存储的值，满足 BST 性质。不过堆性质不再通过旋转维护，而是通过合并维护，实现平衡。而且相同 val 有多个点，不再通过 cnt[] 记录。

核心操作

1. split

拆分操作把一棵以 p 为根的 Treap 拆成两棵 Treap，满足左树所有值 <= d，右树所有值 > d。同时，因为原树满足堆性质，新的两棵树也满足堆性质。

2. merge

合并操作把两棵 Treap x，y 合并成一个 Treap，需要满足 x 中所有元素小于 y 中所有元素，合并的原则是满足堆性质。这里满足的是小根堆。

插入一个值为 d 的节点

FHQTreap 的插入非常有意思，把树先按照 d split成 x 和 y，把新节点作为单独的树 z，先合并 x，z 到 x，再合并 x，y 到 root

删除一个值为 d 的节点

同样很有意思。分三棵树 x，y，z，分别为值 < d 的节点， = d 的节点， > d 的节点。合并 z 的左右儿子到 z，就丢弃了 z 的根节点，也就是我们要删除的值。再把 x，y，z合并到 root

查询一个数 d 的排名

非常谔谔，把树拆成 < d 和 >= d，d 的排名就是 < d 的树的 siz + 1，再合并回去

进阶操作

可持久化

可持久化平衡树用 FHQ 做也很舒服，只需要略加改动，把历史状态存起来，需要注意的是 remove 的时候如果直接改动原状态是坏的，所以 split 的时候要新建节点去搞。

区间维护

例如反转区间 [L, R]，可以 split 出三棵树 [1, L - 1], [L, R], [R + 1, N]。给 [L, R] 打上 tag，再 merge 回去。需要注意的的是要像线段树一样经常 pushdown，而且 split 不是按照权值分了，而是按照 siz 分。
